MaV / Token Architecture Dev Reference
3-Layer Token Architecture

MaV tokens follow a strict three-layer hierarchy: primitives define raw values, semantic tokens assign roles, component tokens scope to specific UI elements. No component should reference a primitive directly.

Layer 0
Primitive
Raw hex values. Single source of truth. Never reference in component code — only through semantic aliases.
--bc-primary-light: #352eff
--bc-polipurple-500: #6a35ff
--bc-mono400-l: #999999
--bc-success: #629c28
Layer 1
Semantic
Role-based aliases. Theme-aware — flip between light/dark by reassigning primitive references, not by duplicating CSS.
--mav-primary: var(--bc-primary-light)
--mav-bg: var(--bc-white)
--mav-fg: var(--bc-black)
--mav-success: var(--bc-success)
Layer 2
Component
Scoped overrides for specific components. Can remap semantic tokens without touching global values.
--btn-bg: var(--mav-primary)
--btn-fg: var(--mav-on-primary)
--input-border: var(--mav-border)
--badge-bg: var(--mav-surface)

Source Files

Token definitions live in three JS files in the design system repo. The CSS is generated from these — never edit the CSS directly.

🎨
baseColors.js
All primitive hex values. Brand palette, extended color ramps (poliPurple, orangeSunshine, pinkPong, manna), mono scale, system colors.
export const baseColors export const monoScale export const systemColors
🔗
colors.js
Semantic token mappings — links primitives to role-based names. Defines both light and dark mode variants. Single file controls theme switching.
export const lightTheme export const darkTheme
🧩
colorsTokens.js
Component-scoped token overrides. Maps semantic tokens to component-specific CSS custom properties. Used by the token generator.
export const buttonTokens export const inputTokens export const badgeTokens
